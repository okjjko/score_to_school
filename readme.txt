# 高校录取分数采集系统 —— 课程实训报告

> **声明（职业规范声明）**
> 本次爬取的数据仅用于课程实训与学习研究，未用于任何商业用途。抓取过程中已严格采取限速与缓存措施（每处理 10 所学校暂停 30 秒、请求间随机延迟、对已抓数据建立本地缓存二次运行秒级跳过），**未对目标网站服务器造成压力**。所调用的均为目标平台公开的开放接口，频率远低于正常浏览行为，已遵守其使用规范与相关法律法规。

---

## 一、项目概述

### 1.1 项目背景

高考志愿填报是每位考生和家长面临的重要决策。目前市面上虽然有不少志愿填报辅助工具，但大多数要么收费高昂，要么数据不够透明。对于普通考生而言，最核心的诉求其实很朴素：**"我的排名/分数，能上哪些学校的哪些专业？"**

为了解决这个问题，我希望构建一个工具：输入自己的省份、年份、目标专业和位次（排名），系统自动遍历全国高校，筛选出录取位次与考生位次接近的专业，从而给出一份"冲稳保"式的匹配学校清单。

**数据价值**：录取分数与位次数据可以帮助考生快速定位目标院校范围，避免在数千所学校中盲目查找；同时积累多年的分数数据后，还能进一步做趋势分析，预测下一年分数线走势。

> **关于数据来源的说明（项目缘起）**
> 在项目前期选型阶段，我尝试了多个数据平台（如各类高考资讯网、百科类数据、政府公开统计等），但普遍存在以下问题：① 数据以 HTML 静态页面展示、字段分散、解析成本高；② 数据不完整，许多学校缺失专业级分数；③ 部分平台有较强的反爬或需登录；④ 字段口径不一，难以统一处理。
> 经过反复对比，最终发现「掌上高考」平台对外提供了较为规整的 **JSON 接口**，数据覆盖全国高校、粒度可达专业级、且包含最低分与最低位次。但网上找到的开源参考实现 `score_to_school` 功能较为基础，仅支持命令行、无断点续传、无数据缓存、无可视化。因此我**以该项目为起点加以深度改造**：新增了 SQLite 双库（缓存库 + 进度库）、可中断的限速与断点续传机制、一元线性回归分数预测，以及一套完整的 Vue 3 + FastAPI 可视化 Web 界面，最终形成了本系统。

### 1.2 项目目标

希望爬取并处理的具体字段：

| 字段                    | 说明                       |
| ----------------------- | -------------------------- |
| 学校名称 / 学校 ID      | 高校唯一标识               |
| 专业名称（spname）      | 具体录取的专业方向         |
| 最低分（min）           | 该专业录取最低分           |
| 最低位次（min_section） | 该专业录取最低位次（排名） |
| 年份 / 省份             | 数据维度                   |

最终按考生位次（差距 < 3000）筛选匹配学校，输出 JSON 排行结果。

## 二、技术选型与设计

### 2.1 请求库

**选用 `requests`**。

理由：目标平台的数据是通过 **Ajax 异步加载的 JSON 接口**返回的（接口地址 `https://api.zjzw.cn/web/api/`），返回值本身就是结构化 JSON，不需要渲染页面、不需要执行 JavaScript。因此没有必要使用 Selenium / Playwright 这类重量级浏览器自动化工具，`requests` 足以胜任，且更轻量、更易控制请求频率。

### 2.2 解析库

**直接使用 Python 内置的 `json` 模块解析**。

理由：接口返回的是标准 JSON（`response.json()`），无需 BeautifulSoup / XPath / 正则等 HTML 解析手段。JSON 解析天然支持嵌套取值（`.get("data").get("item")`），代码简洁可靠。这与前期其他平台"HTML 里捞数据"的痛苦体验形成鲜明对比，也是最终选择该数据源的重要原因。

### 2.3 存储方式

采用 **三层数据落地**：

| 存储介质                    | 用途                                                         |
| --------------------------- | ------------------------------------------------------------ |
| `score.db`（SQLite 缓存库） | 缓存每个 (学校, 省份, 年份) 的原始 API 响应，二次运行直接命中、秒级跳过，避免重复请求触发风控 |
| `progress.db`（进度库）     | 断点续传，记录每个任务已处理的学校列表与中间结果，中断后可恢复 |
| `*.json`（结果文件）        | 最终按学校聚合的匹配结果，命名如 `2024_生物科学_学校分数排行.json` |

### 2.4 项目结构（模块划分）

```
score_to_school/
├── main.py                    # 入口：读取 config.json，调用 process()
├── process.py                 # 核心编排：遍历学校→取分→过滤→聚合→断点续传/限速
├── guess.py                   # 一元线性回归，预测下一年分数（含散点+拟合线）
├── config.json                # 运行配置（省份/专业/年份/位次等）
├── schoolid.json              # 全国高校名称↔ID 映射表（2931 所）
├── Get_informations/
│   ├── GetScore.py            # 获取单校分数（带 SQLite 缓存 + 分页）
│   ├── Getpage.py             # 底层 HTTP 请求封装 + 风控异常
│   ├── Get_province_id.py     # 省份名 → 数字 ID 解析
│   └── Get_school_id.py       # 从 schoolid.json 加载学校列表
├── web/                       # FastAPI 后端（任务化、轮询、持久化恢复）
└── frontend/                  # Vue 3 可视化前端（配置/进度/结果/预测/历史）
```

分工说明：`Getpage` 只管发请求和识别风控；`GetScore` 在其上加缓存与分页；`process` 负责整体编排（遍历、过滤、限速、续传）；`main` 与 `config.json` 提供配置入口。职责逐层向上，单向依赖。

## 三、爬取过程详解（重点）

### 3.1 网站分析

#### 3.1.1 目标 URL 规律（接口与翻页逻辑）

核心数据接口为：

```
https://api.zjzw.cn/web/api/?uri=apidata/api/gk/score/special
       &school_id={学校ID}
       &local_province_id={省份ID}
       &year={年份}
       &page={页码}
       &size=10
```

- **翻页逻辑**：固定每页 `size=10` 条，通过 `page` 参数递增翻页。当某页返回不足 10 条时，说明已到末页，停止翻页。
- 辅助接口：省份 ID 配置（`config/81004.json`）、学校 ID 列表（`school_code.json`，本地化为 `schoolid.json`）。

#### 3.1.2 数据获取方式

数据通过 **Ajax 异步加载的 JSON 接口**返回。页面本身是前端单页应用，真正的数据藏在 XHR/Fetch 请求里。这意味着我们直接请求该 JSON 接口即可拿到结构化数据，绕过了 HTML 渲染与解析环节。

#### 3.1.3 Robots 协议检查与合规性

目标数据接口所属平台为商业志愿填报服务，其开放接口未设置明确的禁止爬取声明。本项目在抓取中严格遵守"礼貌策略"：

- 请求频率被严格限制（详见 3.3），远低于正常人连续浏览的强度；
- 单次抓取后所有原始数据缓存到本地，重复实验不再访问服务器；
- 仅用于个人学习研究，不二次分发、不作商业用途。

>

### 3.2 反爬虫策略与应对

| 反爬手段               | 本项目应对方案                                               |
| ---------------------- | ------------------------------------------------------------ |
| **User-Agent 校验**    | 设置请求头，将 UA 伪装为百度爬虫：`Mozilla/5.0 (compatible; Baiduspider-render/2.0; ...)`，规避基础 UA 检测 |
| **频率限制 / IP 风控** | 三重限速：① 每次请求间 `time.sleep(random.uniform(0.1, 0.3))` 随机延迟；② **每处理 10 所学校暂停 30 秒**；③ 触发"请求过多"风控时，自动 **等待 10 分钟**后重试（不更新进度，保持可恢复） |
| **风控识别**           | 在 `Getpage.py` 中检测响应文本含"请求过多，请稍后再试"时，抛出明确的 `RateLimitError`，由上层 `process` 的 `except` 捕获并进入重试等待 |
| **重复请求浪费**       | 建立 `score.db` 缓存，命中缓存则**完全跳过延迟**，重跑历史数据秒级完成 |

```python
# 核心限速逻辑（process.py 节选）
if not from_cache:              # 仅对真实 API 请求限速，缓存命中直接跳过
    times += 1
    if times % 10 == 0:
        _interruptible_sleep(30, cancel_event, ...)   # 每10所暂停30秒
    else:
        time.sleep(random.uniform(0.1, 0.3))          # 请求间随机延迟
```

### 3.3 数据清洗

主要清洗规则（在 `process.py` 中完成）：

1. **学校类型过滤**：跳过校名含「学院」「职业」的学校，聚焦本科院校；
2. **专业名称匹配**：仅保留专业名（`spname`）包含目标关键字（如「生物科学」）的记录；
3. **专项计划处理**：默认排除含「专项」字样的国家/地方专项计划专业，仅在配置 `ethnic_minority=true` 时保留（针对少数民族等特殊情况）；
4. **位次筛选**：仅保留最低位次与考生位次差距 < 3000 的专业（"冲稳保"范围）；
5. **类型转换**：最低位次 `min_section` 若为字符串则 `int()` 转为整型，转换失败则跳过该条；
6. **去重**：每个 (学校, 省份, 年份) 的原始数据在 `score.db` 中以联合主键去重，断点续传以任务 ID 去重。

## 五、遇到的问题与解决方案

### 问题 1：找不到合适的数据源（前期最大障碍）

**现象**：项目前期尝试了多个平台，数据要么散落在 HTML 中、要么字段不全、要么反爬太强，迟迟无法稳定获取全国专业级分数数据。

**解决**：广泛对比后锁定「掌上高考」的 JSON 接口，并发现开源项目 `score_to_school` 已实现基础抓取。以其为起点二次开发，在保留核心抓取逻辑的同时，补全了缓存、断点续传、预测与可视化等工程化能力。

### 问题 2：抓取中途触发风控（"请求过多，请稍后再试"）

**现象**：连续请求一段时间后，接口返回"请求过多"提示，原版代码在此处因 `data` 未定义直接崩溃（`UnboundLocalError`）。

**解决**：

- 在 `Getpage.py` 中显式检测风控文本，抛出语义清晰的 `RateLimitError`；
- 在 `process.py` 中用 `except` 捕获，**等待 10 分钟后重试**，且重试前**不更新进度**，保证中断点可恢复。

### 问题 3：程序运行一半中断、重头再来

**现象**：全国近 3000 所学校、单次完整抓取耗时极长，一旦断电/关机/误关，已处理的学校全部丢失。

**解决**：引入 `progress.db` 断点续传机制。每处理完一所学校立即落盘（`processed_schools` + `results`），下次启动自动读取已处理集合，跳过已完成学校，从中断点无缝续传。

### 问题 4：重复请求浪费、二次运行依然缓慢

**现象**：调参重跑时，已抓过的学校仍会重新请求，既慢又容易触发风控。

**解决**：建立 `score.db` 缓存，按 (学校, 省份, 年份) 去重。`getScore` 返回 `(数据, from_cache)` 标记，**缓存命中时完全跳过限速延迟**，重跑历史数据秒级完成，只有真正缺失的数据才走 API + 限速。

### 问题 5：多线程下 SQLite 报错 `check_same_thread`

**现象**：原版 `GetScore` 使用模块级全局数据库连接，迁移到 Web 后端的子线程中执行时首调即报错。

**解决**：将全局连接改为**每次调用创建独立连接**（`_get_conn()`），并开启 WAL 模式，保证多线程安全。

### 问题 6：HTTP 请求超时、任务无法在前端长连接等待

**现象**：单次抓取耗时数小时，前端 HTTP 同步请求必然超时。

**解决**：采用「**任务后台化 + 轮询**」架构：`POST /api/task/start` 起后台线程立即返回 `task_id`，前端每 2 秒轮询进度接口，含状态、已处理/总数、匹配数、剩余秒等。配合可中断 sleep（`_interruptible_sleep` 每秒检查取消信号）实现暂停/取消，并通过 `task_state` 表持久化，实现**浏览器刷新与服务重启后的任务恢复**。

## 六、总结与展望

### 6.1 项目收获

1. **爬虫原理理解加深**：深刻体会到"接口型数据（JSON）远比页面型数据（HTML）易于抓取"——这也正是前期反复踩坑后选定当前数据源的根本原因；
2. **反爬与礼貌策略**：掌握了 User-Agent 伪装、限速、风控识别与退避重试等实战技巧，理解了爬虫的"礼貌"既是技术也是规范；
3. **工程化能力**：通过双 SQLite 库（缓存 + 进度）实现了缓存去重与断点续传，体会到"幂等 + 可恢复"设计的重要性；
4. **全栈实践**：将一个命令行脚本升级为 Vue 3 + FastAPI 的全栈可视化应用，涉及任务化、轮询、状态机、持久化恢复等系统设计。

### 6.2 不足与改进

- **并发能力有限**：当前受限于反爬，采用顺序遍历 + 严格限速。如时间充裕，可引入**代理 IP 池**与受控的多线程/分布式调度，在合规前提下提升吞吐；
- **数据维度单一**：目前仅抓分数，可扩展招生计划、选科要求、就业去向等维度，做更全面的志愿推荐；
- **预测模型简单**：一元线性回归仅考虑时间趋势，未纳入考生人数、试卷难度等因素，可尝试多元回归或时序模型提升精度；
- **更严格的礼貌策略**：可进一步接入目标站点频率反馈，动态自适应调整请求间隔。

# score_to_school

从「掌上高考」平台抓取全国高校（共 2931 所）在指定省份、年份、专业的录取分数与位次，按考生位次筛选出匹配院校，输出 JSON 排行结果。

支持 **命令行** 与 **Web 可视化界面** 两种使用方式；并内置 SQLite 缓存、断点续传与一元线性回归分数预测。

---

## 功能特性

- 🎯 **按位次匹配院校**：输入省份、专业、年份与考生位次，自动遍历全国高校，筛选位次差距 < 3000 的专业，输出"冲稳保"式匹配清单。
- ⚡ **SQLite 缓存**：原始响应按 `(学校, 省份, 年份)` 落盘到 `score.db`，二次运行**秒级跳过**已抓数据，不再重复请求。
- 🔁 **断点续传**：长任务可随时中断，`progress.db` 记录已处理学校，重启后从中断点无缝续跑。
- 🛡️ **礼貌反爬**：User-Agent 伪装、请求间随机延迟、每 10 所学校暂停、风控自动退避重试。
- 📈 **分数预测**：`guess.py` 基于历史年份做一元线性回归，预测下一年分数（含散点 + 拟合线）。
- 🖥️ **Web 界面**：Vue 3 + FastAPI 全栈可视化，支持配置抓取、实时进度、结果查看、分数预测、历史续传。

## 快速开始（命令行）

### 1. 安装依赖

```bash
pip install -r requirements.txt
```

### 2. 配置 `config.json`

```jsonc
{
    "province_id": 21,          // 省份：填数字 ID，或填省份名（如 "贵州"）由程序自动解析
    "rank": 12001,              // 你的位次/排名，仅 json 模式需要，会筛选位次接近的专业
    "want": "生物科学",          // 目标专业名称（匹配专业名包含该关键字）
    "year": 2024,               // 年份
    "ethnic_minority": false,   // 是否保留"专项计划"（仅少数民族等特殊情况填 true）
    "output": "json",           // 为真值时输出 JSON 文件（可填 "json" 或 true）
    "subject": ["物理", "生物", "化学"],  // 选科（预留）
    "thread_num": 2             // 线程数：首次抓取某年建议设 2，之后可调高
}
```

> 字段含义也可参见仓库内 `CLAUDE.md` 的详细说明。

### 3. 运行

```bash
python main.py
```

结果输出到 `{年份}_{专业}_学校分数排行.json`，例如 `2024_生物科学_学校分数排行.json`。

## Web 界面使用

除命令行外，项目提供完整 Web 界面。

### 后端（端口 8000）

```bash
# 必须在项目根目录运行，且使用单 worker
python -m uvicorn web.app:app --port 8000
```

> ⚠️ **务必单 worker**：任务状态保存在进程内字典中，多 worker 会导致任务状态分裂。用 `python -m uvicorn`（而非裸 `uvicorn`）确保使用装好依赖的 Python 环境。

### 前端（端口 5173）

```bash
cd frontend
npm install
npm run dev   # 打开 http://localhost:5173
```

Vite 已配置 `/api` 代理到 `http://127.0.0.1:8000`，前后端联调无需额外处理跨域。

### 界面功能

- **配置抓取**：填写省份/专业/年份/位次，一键启动抓取，实时查看进度（已处理/总数、匹配数、当前学校、剩余倒计时）。
- **结果**：查看匹配院校与专业的最低分、最低位次。
- **预测**：基于历史数据回归预测下一年分数，绘制散点 + 拟合线。
- **历史**：查看可续传的断点任务，选择续传项一键恢复。

任务支持暂停 / 取消，且状态落盘——**浏览器刷新或服务重启后任务不丢失**。

## 数据库说明

| 文件          | 用途                                                         |
| ------------- | ------------------------------------------------------------ |
| `score.db`    | API 响应缓存，按 `(schoolId, provinceId, year)` 去重，避免重复请求 |
| `progress.db` | 断点续传（已处理学校列表 + 中间结果）与任务级状态持久化      |

两者均为 SQLite，运行时自动生成，无需单独安装数据库服务。
